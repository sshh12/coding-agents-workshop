The problem

Agents crush greenfield.
They hit a wall on real codebases.

  • You write more docs, bigger CLAUDE.mds, longer prompts
  • The trap: bloated rules files and longer prompts that drown the agent in noise
  • The reframe: redesign the codebase for the agent
  • The mental model: 10,000 interns with 1 week of context — design for that

Dimension 1

Rules File & Agent Config / the 200-line rule

Include

  • Exact commands: pytest, npm run lint, python manage.py check
  • Repo map: where entrypoints, configs, tests live
  • Definition of done: what must be true before agent stops
  • Hard boundaries: "do not touch tests", "ask before adding deps"
  • CLI-first — every agent action must be a shell command, not a GUI click

Exclude

  • Meeting notes, changelogs, or architecture docs
  • General coding philosophy
  • Anything that belongs in a linter config or hook

Beyond CLAUDE.md: the .claude/ directory

  • settings.json: Tool permissions (allow, deny), environment config
  • Hooks: Auto-lint after edits (PostToolUse), block dangerous commands (PreToolUse)
  • rules/ or folder-level CLAUDE.md: Progressive disclosure for monorepos
  • skills/: Repeatable workflows the agent can auto-discover

CLAUDE.md is advisory. Hooks are enforcement. Use both.

06 / 16
Dimension 1 — Personas

Rules File Personas

ML Engineer

  • No CLAUDE.md at all
  • Agent burns tokens inventing conventions every session
  • Guesses test commands, import paths, data directory layout

Quick win: 30-line CLAUDE.md with test command and "do not modify data pipelines"

Monorepo Lead

  • 600-line root CLAUDE.md covering every package
  • Agent in packages/billing never sees billing-specific rules
  • Root noise drowns the signal for every sub-team
  • CI bottlenecked by build graph coupling — agent waits 45 min for feedback

Quick win: Trim root to 50 lines, add per-package .claude/rules/

Progressive disclosure. Root file = navigation. Package files = local rules.

Dimension 2

File Organization / five patterns

  1. Every output is a prompt errors and CLI output guide the agent's next step
  2. Self-documenting code header comments with purpose + related files, --help flags, not external wikis
  3. Predictable interfaces typed configs, enums over strings, conventions the agent already knows
  4. Greppable names descriptive file and function names are the agent's search index
  5. Workflows, not concepts co-locate by feature, not by layer

Single-concern files (split when concerns mix, not at an arbitrary line count). Descriptive names. Feature-grouped folders.

08 / 16
Dimension 2 — Personas

File Organization Personas

Backend Dev

  • 600-line god file mixing routes, models, helpers, and templates
  • Agent greps "tags" and gets 40 hits across 5 concerns
  • Every change risks breaking unrelated features in the same file

Quick win: Split by feature (tags/routes.py, tags/models.py)

Mobile Dev

  • Xcode groups are invisible to filesystem-based agents
  • 200 Swift files flat in one directory
  • Agent has no structural signal — picks wrong file half the time

Quick win: Mirror IDE groups as real folders, add module README

If a human needs an IDE to navigate it, an agent definitely cannot.

Test & Verification / the confidence spectrum

Types
Unit
Integration
E2E
Visual
fast, cheap, narrow slow, expensive, comprehensive

Five principles

  1. Baseline tests are immutable. Protect existing tests from agent modification.
  2. Agents write new tests, test-first. TDD for agents: write the failing test, then implement.
  3. Hooks close the feedback loop. PostToolUse hooks auto-run lint, types, and tests after every edit.
  4. Type the world. Strict types, Pydantic schemas, enums over strings — static analysis is the cheapest trust lever.
  5. Human validation is a speed limit. Automate verification, reserve humans for judgment.

Agent edits file → PostToolUse fires → lint + type-check + pytest → errors fed back → agent self-corrects → repeat

10 / 16
Dimension 3 — Personas

Testing Personas

Frontend Dev

  • No objective verification — quality is "does this look right?"
  • Agent changes a CSS class and has no way to know it broke the layout
  • PR reviews catch regressions days later

Quick win: Playwright screenshots or Storybook snapshots as pass/fail gates

Platform Engineer

  • No terraform validate in CI — destructive ops ungated
  • Agent runs apply on a bad plan and deletes a production database
  • No guardrails between "looks right" and "ships to prod"

Quick win: validate + plan as test commands, PreToolUse hooks block apply/delete

Every dimension needs a command that returns pass/fail. No command = no verification.

Watch out

The Anti-Patterns

PatternWhat happensFix
Test-gamingAgent runs 30 minutes, reports done. App won't load — it deleted most existing tests to make its code "pass"Protect baseline tests with hooks
Implementation-mirroringAgent writes tests that mirror its own implementation. Code is wrong? Test is wrong too — and both look greenRequire behavior-based tests
Rule-gamingLinter flags a violation. Agent greps the monorepo, finds the allowlist config, and modifies it to skip the checkLock config files
No feedback loopAgent codes for 5 minutes straight without running anything. By the time you check, 40 errors have piled upPostToolUse hooks auto-run verification
Bloated context17KB of rules fills 125,996 of 125,999 context tokens. Agent spins with no output — completely frozen200-line limit, use .claude/rules/ for progressive disclosure
Advisory-only rulesCLAUDE.md says "do not modify tests." Agent ignores it, rewrites your test suite, ships broken codeEnforce with hooks and settings.json deny rules
